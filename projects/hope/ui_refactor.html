/ Some Problems

Super loooooong to scroll

To go through the entir seemingly linear and straightflow, frequent back and forth scrolling is often needed

Context is ignored

Timing, frequency and conditions of how and when to use a specific block / field are ignored by the UI. (e.g. the note feature is intensively used throughout the process but is placed in the middle of the UI)

The layout doesn‚Äôt scale...

The current UI is built and expanded by laying blocks in linear order. With increasing complexity of claim structure, the layout ended up lengthy and hard to use

/ Areas to Work on

Blocks with High Usage Frequency

If a block is frequently used. The placement of it should not be linearly fixed in any part of the main UI, but should be decoupled instead. It can either placed in a different scroll view or in a modal

Blocks with Low Usage Frequency

If a block is only used once, it‚Äôs placement should be minimised or even hidden, but only be called when in need




Information Hierarchy

Relative information should be used together and important informaiton should be highlighted. In the above case, mandatory fields should be highlighted and optional blocks should be separated into a more obvious placement

Control Efficiency

Keyboard navigation is considered in the UI revamping to allow key actions to be accessed without locating with cursor

/ The Outcome

The New Revamped UI

The new UI divides the claim form into separate scroll columns. With the larger one for the mandatory actions in sequential order and smaller one for optional actions. And for less actions that are frequently used or not binded into the linear flow, it is being hidden in a modal
