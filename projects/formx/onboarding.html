/ Basic Design Concept

From Receive to Resolve

The underlying concept is simple. The load balancer listens to changes in both the global queue (all tasks) and each local queue (assigned tasks), it will then get tasks from the global queue and pend them into each local queue based on a set of rules that will be further elaborate below




/ Logic Layer One

Properties Matching

Each task has a set of properties, i.e. market info. The load balancer compares the tasks‚Äô properties with each CS‚Äôs expertises to determine which task goes where. If no perfect fit is online, the load balancer will then look for the next best option. Tasks won‚Äôt be assign to CS that is offline or from a different market. Any task not taken in the global queue will appear in the ‚Äòunassigned‚Äô tab




/ Logic Layer Two

Response Time Optimization

In most cases, tasks will be distributed evenly to all CS online. As response time is the utmost metrics to observe, load balancer will re-assign tasks from CS with longer queue to those with shorter to make sure that a task is taken care ASAP



Other Supporting Features

Some little extra touches

While we make sure the balancer is smart enough to do it‚Äôs job. It is always a good idea to reserve rooms for manual actions for a more humane and flexible UX. Below is a few examples that comes together with / on top of the load balancing

Lurker‚Äôs View

It is completely transparent to know who is working on what and how many. The email on each task listing indicates who is watching that task, the thumbnail indicates the current task a teammate is working on.

Personal Watchlist

By default, tasks on a local queue will be re-assigned by check-out. However for complex claims that cannot be resolved instantly, the CS can add it to a personally watchlist so that the tasks is now subscribed by the CS account

Re-assign to Expert

The design assumes the team knows a task better than the load balancer. Flexibility for manually re-assigning tasks to a better hand is always available
